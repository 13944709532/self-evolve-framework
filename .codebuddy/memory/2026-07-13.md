- 用户又发了第二个 token（`npm_YZuR...`）重试，仍 403 同样缺 bypass 2FA（两个 token 连续失败，几乎可确定生成时都漏勾 Bypass 2FA 开关；账号 2FA 模式应为 "Authorization and writes"，普通 token 一律被拦）。
- 放弃 token 路线，回退交互 OTP 方案：已 `npm config delete //registry.npmjs.org/:_authToken` 清掉 `.npmrc` 官方源 token 行（保留镜像源 registry 行）。本地终端须先 `npm adduser --registry https://registry.npmjs.org`（输 OTP 登录），再 `npm publish --registry https://registry.npmjs.org`（交互提示 Enter OTP 输 6 位码）才能发布 1.6.1。
- 教训：`.npmrc` 里只要有无 bypass 的 token，本地 `npm publish` 就不会走交互 OTP，直接 403；务必清空 token 让其回退交互登录。
- **重要更正（推翻上条 `--otp` 结论）**：经实测，账号 2FA 为 "Authorization and writes" 模式时，`.npmrc` 中任何普通 token（含 `npm adduser` 写回的登录 token，如 `npm_PkSX...`）都会让 `npm publish` 走 bearer token 路径，registry 对该无 bypass 2FA 的 token **直接返回 403 且不触发 OTP 提示**；npm 11.17 下 `--otp` 参数对 token 路径**完全无效**。故"本地 `--otp` 发布"在此账号模式下**不可行**（之前建议有误）。真正可行的仅两条：① 生成**勾选了 Bypass 2FA** 的 Granular Access Token（npm 网页会警告安全风险，对个人发布可接受），配入 `.npmrc` 后 `npm publish` 无需 OTP；② **Trusted Publishing (OIDC)**，已建 `.github/workflows/publish.yml`，npm 包配 Trusted Publisher + 打 tag 即自动发布。用户已倾向 Trusted Publishing。
- 最终决策：用户在 npm 官网生成 Bypass 2FA token 时，页面强烈警告 "There are security risks... please use Trusted Publishing instead"，但因手机 Authenticator 无法添加 npm（2FA 验证又卡在 Trusted Publisher 网页），**回退 Bypass 2FA Granular Token 路线**。已生成 `npm_XBFJq...Qss`（勾选 Bypass 2FA），写入 `.npmrc`，**成功发布 `1.6.1` 到官方 registry**（`npm view ... version` 返回 1.6.1）。
- 核心结论：账号 2FA 为 "Authorization and writes" 模式时，仅 `_authToken` 为带 Bypass 2FA 的 Granular Token 才能发布；普通 token（含 adduser 登录 token）+ `--otp` 均无效（registry 对无 bypass token 直接 403 不触发 OTP）。`.npmrc` 现已有有效 bypass token，后续 `npm publish --registry https://registry.npmjs.org` 直接可发，无需 OTP。
- Trusted Publishing 的 `.github/workflows/publish.yml` 已建且已提交推送，作为长期方案备选（需解决 2FA 验证网页关）。
